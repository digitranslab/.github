### Contributing Documentation

1. **Fork the repository** to your GitHub account
2. **Clone your fork** locally
3. **Create a branch** for your changes (`git checkout -b feature/your-feature-name`)
4. **Make your changes** following our style guidelines
5. **Commit your changes** with clear, descriptive messages
6. **Push to your fork** and submit a pull request

1. **Ensure your PR addresses a single concern** - keep changes focused and atomic

- [ ] You have read and followed the style guidelines

## ğŸ“ Style Guidelines

### Language and Spelling

- Use **British English** throughout all documentation
- Examples: colour (not color), organisation (not organization), behaviour (not behavior)
- Use clear, concise language accessible to international readers

### Markdown Formatting

- Use ATX-style headers (`#`, `##`, `###`)
- Include a table of contents for documents exceeding 200 lines
- Use fenced code blocks with language identifiers
- Use relative links for internal references

### Badges and Visual Elements

- **Main badges**: Use `style=for-the-badge`
- **Inline badges**: Use `style=flat-square`
- **Colour palette**: Purple (#8B5CF6), Blue (#3B82F6)
- Use emojis sparingly for visual hierarchy

### File Naming

- Use UPPERCASE for root-level documentation (README.md, CONTRIBUTING.md)
- Use UPPERCASE with hyphens for multi-word files (CASE-STUDIES.md)
- Use lowercase for files within docs/ directories

### Content Structure

- Start each document with a clear title and brief description
- Include navigation links to parent and related documents
- Use tables for structured information
- Use collapsible sections for detailed content
